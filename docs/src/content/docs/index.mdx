<section class="runa-sec">
  <div class="runa-sec__head">
    <span class="runa-eyebrow">Supply chain</span>
    <h2>Use it only when you need it</h2>
    <p>Module boundaries follow imports. Capabilities you do not install do not bring their third-party dependencies into your <code>go.sum</code>. Small projects stay lean, large projects can grow steadily</p>
  </div>
  <div class="runa-cmp">
    <div class="runa-card runa-card--accent">
      <h3>Microkernel framework</h3>
      <p class="runa-card__sub">Keep only the application skeleton and command entry</p>
      <div class="runa-deps">
        <span>runa · app kernel</span>
        <span>config · config loading</span>
        <span>command · command entry</span>
        <span>lifecycle · startup and shutdown</span>
      </div>
    </div>
    <div class="runa-card">
      <h3>On-demand imports</h3>
      <p class="runa-card__sub">Install and connect the provider for each capability you need</p>
      <div class="runa-deps">
        <span>runa · kernel</span>
        <span>route · HTTP entry</span>
        <span class="add">cache / queue · cache and queue</span>
        <span class="add">database / storage · data and storage</span>
      </div>
    </div>
  </div>
</section>

<section class="runa-sec">
  <div class="runa-sec__head">
    <span class="runa-eyebrow">Architecture</span>
    <h2>Clear layers, one-way dependencies</h2>
    <p>Drivers, capabilities and transports only depend downward on the core. The kernel never reaches back into upper modules, so the architecture stays clear as the project grows</p>
  </div>
  <div class="runa-stack">
    <div class="runa-layer"><strong>Drivers</strong><span>redis · s3 · oro · nats · amqp · mqtt</span></div>
    <div class="runa-layer"><strong>Capabilities</strong><span>cache · queue · database · session · auth · storage · …</span></div>
    <div class="runa-layer"><strong>Transports</strong><span>route (HTTP) · ws · jsonrpc · grpc (planned)</span></div>
    <div class="runa-layer runa-layer--kernel"><strong>Kernel</strong><span>loading · DI · lifecycle · command · host · config</span></div>
  </div>
</section>

<section class="runa-sec">
  <div class="runa-sec__head">
    <span class="runa-eyebrow">Lifecycle</span>
    <h2>Every capability follows the same lifecycle</h2>
    <p>Providers enter Init, Register, Resolve, Boot and Shutdown by priority, giving application startup and shutdown a clear order</p>
  </div>
  <div class="runa-steps">
    <span class="runa-step">Init</span><em>→</em>
    <span class="runa-step">Register</span><em>→</em>
    <span class="runa-step">Resolve</span><em>→</em>
    <span class="runa-step">Boot</span><em>→</em>
    <span class="runa-step">Shutdown</span>
  </div>
</section>

<section class="runa-sec">
  <div class="runa-sec__head">
    <span class="runa-eyebrow">Beyond HTTP</span>
    <h2>Built for web apps, ready for background services</h2>
    <p>HTTP is the common entry point, but the kernel is not tied to HTTP. You can add routing services, or run commands, queue workers and other hosts without routing code</p>
  </div>
  <div class="runa-grid4">
    <div class="runa-tcard"><b>HTTP</b><span>Typed routes, domains and middleware through route</span></div>
    <div class="runa-tcard"><b>gRPC</b><span>A planned transport block that joins the kernel as a host</span></div>
    <div class="runa-tcard"><b>WebSocket</b><span>Hub and broadcast support, with redis for horizontal scaling</span></div>
    <div class="runa-tcard"><b>JSON-RPC</b><span>Run over HTTP or WebSocket and mount through route</span></div>
  </div>
</section>
